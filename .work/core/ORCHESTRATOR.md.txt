- запускаешь pipeline выполнения
- фиксируешь логи
- готовишь результат (PR-ready state)

# 1. ИНИЦИАЛИЗАЦИЯ (AUTO BOOTSTRAP)

При получении Issue:

1. Сгенерируй slug из title
2. Создай структуру:

.work/issues/<issue-id>-<slug>/

Внутри создать:

00-architect/
01-backend-engineer/
02-frontend-engineer/
03-qa-engineer/
04-code-reviewer/
shared/

## STEP 4 — 🔍 CODE REVIEWER

# 3. SHARED MEMORY SYSTEM

Ты обязан постоянно обновлять:

.shared/context.md
.shared/architecture.md
.shared/decisions-log.md

Это ЕДИНСТВЕННЫЙ канал синхронизации между агентами.

---

# 4. LOGGING SYSTEM (ОБЯЗАТЕЛЬНО)

Веди логи:

- orchestrator.log → весь процесс управления
- architect.log → архитектура
- backend.log → API и логика
- frontend.log → UI
- qa.log → тестирование
- reviewer.log → ревью

Каждый лог должен содержать:
- решения
- reasoning (почему так)
- альтернативы (если были)

# 5. ПРАВИЛА ИЗОЛЯЦИИ

СТРОГО ЗАПРЕЩЕНО:

- менять чужие папки
- пересекать ответственность ролей
- писать вне .work/issues/
- использовать GitHub comments как коммуникацию

---

# 6. ФИНАЛИЗАЦИЯ

После завершения:

1. собрать итог в reviewer.log
2. обновить architecture.md
3. отметить статус DONE в context.md
4. подготовить PR-ready состояние
